## 数据模型总览

本项目共包含 **10 张核心业务表**，分为以下 5 大业务模块：

| 模块 | 表名 | 业务含义 | 记录数预估 |
|------|------|---------|-----------|
| **用户模块** | `tb_user` | 用户基础信息 | 百万级 |
| | `tb_user_info` | 用户详细信息 | 百万级 |
| **商户模块** | `tb_shop_type` | 商户类型（分类） | 10-20条 |
| | `tb_shop` | 商户信息 | 万级 |
| **优惠券模块** | `tb_voucher` | 优惠券（普通券） | 千级 |
| | `tb_seckill_voucher` | 秒杀券 | 百级 |
| | `tb_voucher_order` | 优惠券订单 | 百万级 |

| | `tb_blog_comments` | 笔记评论 | 千万级 |

## 核心业务模块

### 1. 用户模块 (User Module)

#### 1.1 `tb_user` - 用户基础表

**业务含义**：存储用户登录凭证和基础信息，是系统的核心身份表。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 用户ID（主键） |
| `phone` | VARCHAR(11) | UNIQUE, NOT NULL | 手机号（登录凭证，唯一索引） |
| `password` | VARCHAR(128) | NULL | 密码（加密存储，验证码登录时可为空） |
| `nick_name` | VARCHAR(32) | NULL | 昵称（默认随机生成） |
| `icon` | VARCHAR(255) | NULL | 头像URL |
| `create_time` | TIMESTAMP | NOT NULL | 创建时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ `phone` 字段有**唯一索引**，防止重复注册
- ✅ `password` 可为空，支持验证码登录（无需密码）
- ✅ 高频查询字段（登录时必查），建议在 Redis 中缓存

---

#### 1.2 `tb_user_info` - 用户详情表

**业务含义**：存储用户扩展信息，与 `tb_user` 是**一对一关系**（垂直分表）。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `user_id` | BIGINT | PK | 用户ID（与 tb_user.id 一对一） |
| `city` | VARCHAR(64) | NULL | 所在城市 |
| `introduce` | VARCHAR(128) | NULL | 个人介绍 |
| `fans` | INT | DEFAULT 0 | 粉丝数量（冗余字段，提高查询性能） |
| `followee` | INT | DEFAULT 0 | 关注数量（冗余字段） |
| `gender` | TINYINT(1) | DEFAULT 0 | 性别（0:男, 1:女） |
| `birthday` | DATE | NULL | 生日 |
| `credits` | INT | DEFAULT 0 | 积分 |
| `level` | TINYINT(1) | DEFAULT 0 | 会员级别（0-9级，0表示未开通） |
| `create_time` | TIMESTAMP | NOT NULL | 创建时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **垂直分表**：`tb_user` 是热点表（每次登录都查），`tb_user_info` 是低频表，拆分后减少单表大小
- ✅ **冗余字段**：`fans`、`followee` 是统计字段，避免每次查询都 `COUNT(*)`
- ⚠️ **数据一致性**：需要保证 `fans` 与 `tb_follow` 表的数据一致（通过定时任务或触发器维护）

---

### 2. 商户模块 (Shop Module)

#### 2.1 `tb_shop_type` - 商户类型表

**业务含义**：商户分类（如"美食"、"KTV"、"丽人·美发"），用于首页分类导航。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 类型ID |
| `name` | VARCHAR(32) | NULL | 类型名称（如"美食"） |
| `icon` | VARCHAR(255) | NULL | 图标URL |
| `sort` | INT(3) | NULL | 显示顺序（数字越小越靠前） |
| `create_time` | TIMESTAMP | NULL | 创建时间 |
| `update_time` | TIMESTAMP | NULL | 更新时间 |

**关键设计**：
- ✅ **低频变更**：类型数据很少变化，适合缓存（Redis List）
- ✅ **有序展示**：`sort` 字段控制前端展示顺序

**示例数据**：
```
id=1, name="美食", icon="/types/ms.png", sort=1
id=2, name="KTV", icon="/types/KTV.png", sort=2
id=3, name="丽人·美发", icon="/types/lrmf.png", sort=3
```

---

#### 2.2 `tb_shop` - 商户信息表

**业务含义**：存储商户的详细信息，包括地理位置、评分、销量等。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 商户ID |
| `name` | VARCHAR(128) | NOT NULL | 商户名称 |
| `type_id` | BIGINT | NOT NULL, INDEX | 商户类型ID（外键，关联 tb_shop_type.id） |
| `images` | VARCHAR(1024) | NOT NULL | 商户图片（多张用逗号分隔） |
| `area` | VARCHAR(128) | NULL | 商圈（如"陆家嘴"） |
| `address` | VARCHAR(255) | NOT NULL | 详细地址 |
| `x` | DOUBLE | NOT NULL | 经度（用于 GEO 查询） |
| `y` | DOUBLE | NOT NULL | 纬度（用于 GEO 查询） |
| `avg_price` | BIGINT | NULL | 人均消费（单位：分） |
| `sold` | INT | NOT NULL | 销量（冗余字段） |
| `comments` | INT | NOT NULL | 评论数（冗余字段） |
| `score` | INT(2) | NOT NULL | 评分（1-5分，存储时×10，如 4.5分存45） |
| `open_hours` | VARCHAR(32) | NULL | 营业时间（如"10:00-22:00"） |
| `create_time` | TIMESTAMP | NULL | 创建时间 |
| `update_time` | TIMESTAMP | NULL | 更新时间 |

**关键设计**：
- ✅ **地理位置索引**：`x`、`y` 字段用于 Redis GEO 查询（附近商户）
- ✅ **冗余字段**：`sold`、`comments` 避免频繁 `COUNT(*)` 查询
- ✅ **评分存储**：`score` 乘以 10 存储，避免浮点数精度问题
- ✅ **外键索引**：`type_id` 有索引，提高按类型查询性能

**虚拟字段**（`@TableField(exist = false)`）：
- `distance`：计算出的距离（米），不存数据库，仅用于返回给前端

### 3. 优惠券模块 (Voucher Module)

#### 3.1 `tb_voucher` - 优惠券表（普通券）

**业务含义**：存储普通优惠券信息（非秒杀券）。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 优惠券ID |
| `shop_id` | BIGINT | NULL | 所属商户ID（NULL表示平台券） |
| `title` | VARCHAR(255) | NOT NULL | 券标题 |
| `sub_title` | VARCHAR(255) | NULL | 副标题 |
| `rules` | VARCHAR(1024) | NULL | 使用规则 |
| `pay_value` | BIGINT | NOT NULL | 支付金额（单位：分，如200表示2元） |
| `actual_value` | BIGINT | NOT NULL | 抵扣金额（单位：分） |
| `type` | TINYINT(1) | DEFAULT 0 | 券类型（0:普通券, 1:秒杀券） |
| `status` | TINYINT(1) | DEFAULT 1 | 状态（1:上架, 2:下架, 3:过期） |
| `create_time` | TIMESTAMP | NOT NULL | 创建时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **金额存储**：使用 `BIGINT` 存储分，避免浮点数精度问题
- ✅ **类型区分**：`type=1` 的券，会在 `tb_seckill_voucher` 中有对应记录

**虚拟字段**（`@TableField(exist = false)`）：
- `stock`：库存（秒杀券才有，从 `tb_seckill_voucher` 表查询）
- `begin_time`：生效时间（秒杀券才有）
- `end_time`：失效时间（秒杀券才有）

---

#### 3.2 `tb_seckill_voucher` - 秒杀券表

**业务含义**：存储秒杀券的库存和时间信息，与 `tb_voucher` 是**一对一关系**。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `voucher_id` | BIGINT | PK | 优惠券ID（与 tb_voucher.id 一对一，主键） |
| `stock` | INT(8) | NOT NULL | 库存数量（秒杀核心字段） |
| `begin_time` | TIMESTAMP | NOT NULL | 秒杀开始时间 |
| `end_time` | TIMESTAMP | NOT NULL | 秒杀结束时间 |
| `create_time` | TIMESTAMP | NOT NULL | 创建时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **主键设计**：使用 `voucher_id` 作为主键（`IdType.INPUT`），与 `tb_voucher.id` 保持一致
- ✅ **库存字段**：`stock` 是秒杀业务的核心，需要**乐观锁**或**Redis 预减库存**保护
- ✅ **时间控制**：`begin_time`、`end_time` 控制秒杀活动时间窗口

**业务逻辑**：
- 秒杀券的库存会**预加载到 Redis**（`seckill:stock:{voucherId}`）
- 下单时先减 Redis 库存，再异步减 MySQL 库存

---

#### 3.3 `tb_voucher_order` - 优惠券订单表

**业务含义**：记录用户购买优惠券的订单信息。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK | 订单ID（使用雪花算法生成，非自增） |
| `user_id` | BIGINT | NOT NULL | 下单用户ID |
| `voucher_id` | BIGINT | NOT NULL | 购买的优惠券ID |
| `pay_type` | TINYINT(1) | DEFAULT 1 | 支付方式（1:余额, 2:支付宝, 3:微信） |
| `status` | TINYINT(1) | DEFAULT 1 | 订单状态（1:未支付, 2:已支付, 3:已核销, 4:已取消, 5:退款中, 6:已退款） |
| `create_time` | TIMESTAMP | NOT NULL | 下单时间 |
| `pay_time` | TIMESTAMP | NULL | 支付时间 |
| `use_time` | TIMESTAMP | NULL | 核销时间（到店使用时间） |
| `refund_time` | TIMESTAMP | NULL | 退款时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **主键策略**：使用**雪花算法**（`SnowFlakeIDWorker`）生成订单ID，而非数据库自增
  - **原因**：分布式环境下，自增ID 可能冲突；雪花算法保证全局唯一且有序
- ✅ **状态机设计**：`status` 字段记录订单完整生命周期
- ✅ **时间戳字段**：多个时间字段记录订单各阶段时间，便于数据分析和问题排查

**业务约束**：
- **一人一单**：同一用户同一秒杀券只能购买一次（通过 Redis Set 或数据库唯一索引保证）

### 4. 社交模块 (Social Module)

#### 4.1 `tb_blog` - 探店笔记表

**业务含义**：用户发布的探店笔记（类似大众点评的"评价"功能）。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 笔记ID |
| `shop_id` | BIGINT | NOT NULL | 关联的商户ID |
| `user_id` | BIGINT | NOT NULL | 发布用户ID |
| `title` | VARCHAR(255) | NOT NULL | 笔记标题 |
| `images` | VARCHAR(2048) | NOT NULL | 图片（最多9张，逗号分隔） |
| `content` | VARCHAR(2048) | NOT NULL | 文字内容 |
| `liked` | INT(8) | DEFAULT 0 | 点赞数（冗余字段） |
| `comments` | INT(8) | NULL | 评论数（冗余字段） |
| `create_time` | TIMESTAMP | NOT NULL | 发布时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **冗余字段**：`liked`、`comments` 避免频繁统计查询
- ✅ **点赞存储**：点赞数据存储在 Redis SortedSet（`blog:liked:{blogId}`），不直接更新 `liked` 字段
  - **原因**：点赞是高频操作，直接写数据库压力大；Redis 异步同步到 MySQL

**虚拟字段**（`@TableField(exist = false)`）：
- `icon`：发布者头像（联表查询或缓存获取）
- `name`：发布者昵称
- `isLike`：当前用户是否已点赞（从 Redis 查询）

---

#### 4.2 `tb_blog_comments` - 笔记评论表

**业务含义**：笔记的评论系统，支持**二级评论**（一级评论 + 回复）。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 评论ID |
| `user_id` | BIGINT | NOT NULL | 评论用户ID |
| `blog_id` | BIGINT | NOT NULL | 关联的笔记ID |
| `parent_id` | BIGINT | NOT NULL | 父评论ID（一级评论为0） |
| `answer_id` | BIGINT | NOT NULL | 回复的评论ID（一级评论为0） |
| `content` | VARCHAR(255) | NOT NULL | 评论内容 |
| `liked` | INT(8) | NULL | 点赞数 |
| `status` | TINYINT(1) | NULL | 状态（0:正常, 1:被举报, 2:禁止查看） |
| `create_time` | TIMESTAMP | NOT NULL | 评论时间 |
| `update_time` | TIMESTAMP | NOT NULL | 更新时间 |

**关键设计**：
- ✅ **二级评论设计**：
  - `parent_id = 0`：一级评论（直接评论笔记）
  - `parent_id != 0`：二级评论（回复某条评论）
  - `answer_id`：被回复的评论ID（用于显示"回复 @xxx"）

**示例数据结构**：
```
笔记ID=1
├── 评论1 (id=1, parent_id=0, answer_id=0) "这家店不错！"
│   └── 回复1 (id=2, parent_id=1, answer_id=1) "我也觉得！"
└── 评论2 (id=3, parent_id=0, answer_id=0) "价格有点贵"
```

---

#### 4.3 `tb_follow` - 用户关注表

**业务含义**：记录用户之间的关注关系（类似微博的关注功能）。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | BIGINT | PK, AUTO_INCREMENT | 主键 |
| `user_id` | BIGINT | NOT NULL | 关注者ID（谁关注） |
| `follow_user_id` | BIGINT | NOT NULL | 被关注者ID（关注谁） |
| `create_time` | TIMESTAMP | NOT NULL | 关注时间 |

**关键设计**：
- ✅ **关系存储**：同时存储在 Redis Set 中（`follow:{userId}`），提高查询性能
- ✅ **Feed 流**：关注关系用于实现"关注的人的动态"（Feed 流）
- ⚠️ **数据一致性**：需要保证 Redis Set 与 MySQL 表数据一致

**业务场景**：
- **共同关注**：计算两个用户的共同关注（Redis `SINTER` 命令）
- **关注推送**：用户发布笔记时，推送到所有粉丝的 Feed 流（Redis ZSet）

## 关键设计说明

### 1. 主键策略

| 表名 | 主键策略 | 原因 |
|------|---------|------|
| `tb_user` | AUTO_INCREMENT | 用户ID不需要全局唯一性要求 |
| `tb_shop` | AUTO_INCREMENT | 商户ID不需要全局唯一性要求 |
| `tb_voucher_order` | **雪花算法** | 分布式环境下保证全局唯一，且有序 |
| `tb_seckill_voucher` | INPUT (使用 voucher_id) | 与 tb_voucher.id 保持一致 |

### 2. 索引设计

| 表名 | 索引字段 | 索引类型 | 说明 |
|------|---------|---------|------|
| `tb_user` | `phone` | UNIQUE | 登录凭证，唯一性保证 |
| `tb_shop` | `type_id` | INDEX | 按类型查询商户 |
| `tb_voucher_order` | `user_id` + `voucher_id` | **复合唯一索引** | 保证"一人一单"（秒杀场景） |

### 3. 冗余字段设计

**为什么使用冗余字段？**
- **性能优化**：避免频繁 `COUNT(*)` 查询
- **数据一致性风险**：需要通过定时任务或触发器维护

| 表名 | 冗余字段 | 对应统计表 | 维护方式 |
|------|---------|-----------|---------|
| `tb_user_info` | `fans`, `followee` | `tb_follow` | 关注/取关时更新 |
| `tb_shop` | `sold`, `comments` | `tb_voucher_order`, `tb_blog` | 下单/评论时更新 |
| `tb_blog` | `liked`, `comments` | Redis + `tb_blog_comments` | Redis 异步同步 |

### 4. 垂直分表设计

**`tb_user` 与 `tb_user_info` 拆分原因**：
- `tb_user`：高频查询（每次登录都查），字段少，查询快
- `tb_user_info`：低频查询（查看个人主页才查），字段多，数据量大
- **拆分后**：减少 `tb_user` 单行大小，提高缓存命中率

### 5. 金额存储设计

**为什么用 `BIGINT` 存储"分"而不是 `DECIMAL` 存储"元"？**
- ✅ **精度保证**：避免浮点数精度问题（如 0.1 + 0.2 ≠ 0.3）
- ✅ **性能优化**：整数运算比浮点数快
- ✅ **存储节省**：`BIGINT` 比 `DECIMAL` 占用空间小

**示例**：
- 存储：`pay_value = 200`（表示 2.00 元）
- 显示：前端除以 100 显示为 "¥2.00"

### 6. 时间字段设计

**为什么多个时间字段？**
- `create_time`：记录创建时间（不可变）
- `update_time`：记录最后更新时间（自动更新）
- `pay_time`、`use_time`、`refund_time`：记录订单各阶段时间，便于：
  - 数据分析（支付转化率、核销率）
  - 问题排查（订单异常时定位时间点）

## 总结

本项目数据模型设计特点：

1. **垂直分表**：用户表拆分，提高查询性能
2. **冗余字段**：减少统计查询，提高响应速度
3. **主键策略**：订单表使用雪花算法，保证分布式唯一性
4. **金额存储**：使用分存储，避免精度问题
5. **Redis 配合**：热点数据缓存，复杂数据结构（GEO、SortedSet）提升性能
