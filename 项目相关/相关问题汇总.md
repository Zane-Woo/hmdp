# 为什么要使用双层拦截器？

通用拦截器负责刷新token，这是所有网页都需要的，登录拦截器针对特定网址负责权限校验。（职责划分清晰）

对于当前的项目而言，只使用一层拦截器固然可以实现，但是根据逻辑划分拦截器很有助于后续的代码的演进与维护。比如说之后还需要加日志记录之类。如果只写成一层会造成代码逻辑的混杂，冗余，可维护性和可复用性差。

# 为什么要使用ThreadLocal?

主要解决的问题就是参数的上下文传递的问题。因为后续的接口需要使用到**当前请求的用户信息**，**而一个线程对应一个用户的请求，也就是一个用户的ID**，通过获取当前线程的对应的ThreadLocal对应的值即可获取当前线程传递的隐式上下文。当然，也可以通过参数传递，但是必须在方法中添加参数并且需要同步调用（一旦在调用链中的某个环节没有持续传递，上下文就会丢失），不仅无法灵活地异步调用，而且这样代码侵入式很强。

记得当时我被问“有没有什么替代性的方案的时候”，竟然回答可以使用redis存对应的token。这也太蠢了，明明这里的userId是依赖于请求的，也就是依赖于当前线程的，除非你把当前的线程信息存下来，但这样会对redis造成很大压力。其实如果是jwt的话也可以存token，但是就失去jwt无状态的意义了。而且这个项目的其实完全算不上严格意义的token，实际上就是一个session id。

 

# 为什么前面没有使用Redisson而是Redis的分布式锁？

完全可以使用Redisson，但是Redis的分布式锁也完全可以胜任。主要是取决于当前的实际场景，这里是为了读取最新的值到缓存中，但是为了防止大量线程打到db上，所以使用了分布式锁保证最多只有一个线程读db的情况，注意不是改db，所以不会发生超卖之类的情况。

所以即便是发生一个线程持有锁超时被其他线程持有导致两个以上的线程访问也没有关系，因为这种情况发生了读db的线程也还是很少。

不过判断是否为新值然后直接返回还是很有必要的，避免无意义的读db操作

* **为什么db不能大量线程读写，但是redis可以？**
  * db是基于磁盘的，所以处理读写的速度相对于内存慢得多，而且mysql的连接池最大处理的连接数也有限，没有拿到连接的线程只能等待，然后后续的请求不断地堆积，知道mysql拒绝请求或者OOM。

# 这里的shop确实考虑了很多缓存相关的问题，但是为什么同样作为热点数据的voucher却没有考虑这经典的三个问题？

Shop和voucher的读写模式不一样，前者使用的是旁路缓存，缓存中的数据不是单一事实源，所以会出现这个缓存相关的问题，更新数据是先更新db。但是后者是使用了异步缓存写入的策略，将缓存当作单一事实源，所有的写操作都是操作cache（因为将活动相关的所有voucher都提前预热到缓存，并且ttl为整个活动）。然后靠业务端实现异步写入db，这里的db仅仅是一个备份。
